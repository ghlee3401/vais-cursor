### 🔧 fix — 영향 분석 기반 수정

**실행 시작 시 반드시** 아래 인트로를 먼저 출력하세요:

```
---
📋 **VAIS Manager** 입니다.
수정 요청을 분석하고 영향 범위를 파악하겠습니다.
---
```

사용자가 UI나 기능 수정을 요청할 때, **Manager를 통해 영향 범위를 파악**하고 연관된 코드·문서를 일괄 수정합니다.

> **Manager 경유**: fix는 Manager 에이전트를 호출합니다. Manager가 memory 기반으로 크로스-피처 영향을 분석한 후 Tech Lead에게 수정을 지시합니다.

#### Step 0: Manager 호출

Agent 도구로 **manager** 에이전트를 호출합니다:
- 사용자 수정 요청 원문 전달
- Manager가 `.vais/memory.json`에서 과거 의사결정, 피처 간 의존성 조회
- 크로스-피처 영향 분석 후 Tech Lead에게 지시
- 아래 Step 1~6은 Manager → Tech Lead 위임 후 실행되는 흐름입니다.

#### Step 1: 피처 자동 감지

1. 사용자의 수정 요청을 분석
2. **피처 자동 감지**:
   - `.vais/status.json`에서 활성 피처 확인
   - 수정 대상 파일이 어떤 피처에 속하는지 판단
   - 불확실하면 AskUserQuestion으로 확인
3. **피처 레지스트리 로드** (`.vais/features/{피처}.json`)
4. **관련 문서 전체 수집**:
   - 기획서 (`docs/02-plan/{피처}.md`) — 기능 목록, 정책 정의
   - IA (`docs/03-ia/{피처}.md`) — 화면 흐름
   - 와이어프레임 (`docs/04-wireframe/{피처}.md`) — 레이아웃
   - 설계 (`docs/05-design/{피처}.md`) — 컴포넌트 명세, 상태 관리
   - DB 설계 (`docs/05-design/{피처}-db.md`) — 스키마 (있는 경우)

#### Step 2: 영향 분석

수정 요청을 아래 카테고리로 분류하고, 각각에 맞는 영향 범위를 분석합니다:

**수정 유형별 영향 범위:**

| 유형 | 코드 영향 | 문서 영향 |
|------|----------|----------|
| **UI/레이아웃 변경** | 해당 컴포넌트 + 부모/자식 컴포넌트 + CSS | 와이어프레임, 설계 |
| **기능 변경** | 관련 컴포넌트 + API + 상태 관리 | 기획서(기능 목록), 설계, 피처 레지스트리 |
| **정책 변경** | 유효성 검증 + 비즈니스 로직 + API | 기획서(정책 정의), 피처 레지스트리 |
| **데이터 변경** | API + DB 스키마 + ORM + 마이그레이션 | DB 설계, 기획서 |
| **화면 추가/삭제** | 라우팅 + 새 컴포넌트 + 네비게이션 | IA, 와이어프레임, 설계, 피처 레지스트리 |

**분석 결과를 테이블로 출력:**

```markdown
## 🔧 영향 분석

**수정 요청**: "{사용자 요청 요약}"
**수정 유형**: UI/레이아웃 변경
**관련 피처**: login

### 영향 받는 파일

| # | 파일 | 변경 내용 | 영향도 |
|---|------|----------|--------|
| 1 | src/components/LoginForm.tsx | 버튼 위치 변경 | 🔴 직접 |
| 2 | src/components/Header.tsx | 레이아웃 조정 필요 | 🟡 연쇄 |
| 3 | docs/04-wireframe/login.md | 와이어프레임 업데이트 | 🔵 문서 |
| 4 | docs/05-design/login.md | 컴포넌트 명세 수정 | 🔵 문서 |

### 정책 충돌 확인
- ✅ 기획서 정책과 충돌 없음
- ⚠️ "비밀번호 입력 필드는 항상 버튼 바로 위에 배치" 정책과 충돌 가능 → 사용자 확인 필요
```

#### Step 3: 사용자 확인

AskUserQuestion으로 확인:
- "위 영향 분석 결과대로 수정을 진행할까요?"
- 선택지:
  1. "진행해 주세요" → Step 4로
  2. "범위를 좁혀주세요 (코드만 / 문서만)"
  3. "취소"

**정책 충돌이 있으면** 반드시 사용자에게 알리고 확인:
- "기존 정책 '{정책 내용}'과 충돌합니다. 정책도 같이 변경할까요?"

#### Step 4: 체이닝 실행

영향 분석 결과에 따라 **재실행이 필요한 단계를 체이닝으로 실행**합니다. fix가 직접 수정하지 않고, 각 단계의 전문 로직에 위임합니다.

**수정 유형별 체이닝:**

| 유형 | 체이닝 | 예시 |
|------|--------|------|
| UI/레이아웃 변경 | `wireframe:design:fe` | "아이콘 중앙 정렬해줘" |
| 스타일만 변경 | `fe` | "버튼 색상 바꿔줘" |
| 기능 변경 | `plan:design:fe+be` | "로그인에 소셜 로그인 추가해줘" |
| 정책 변경 | `plan:fe+be` | "비밀번호 규칙 바꿔줘" |
| 데이터 변경 | `design:be` | "API 응답 구조 바꿔줘" |
| 화면 추가/삭제 | `plan:ia:wireframe:design:fe+be` | "마이페이지 추가해줘" |
| 전체 흐름 변경 | `plan:ia:wireframe:design:fe+be:check` | "로그인 흐름 전면 재설계" |

**실행 방식:**
1. 영향 분석 결과에서 **가장 상위 영향 단계**를 시작점으로 결정
2. 해당 단계부터 구현 단계까지 체이닝 구성
3. 각 단계는 기존 문서를 **수정 요청 반영하여 업데이트** (신규 생성이 아님)
4. 병렬 가능 구간(`fe+be`)은 자동 적용

#### Step 5: 수정 후 검증

1. **빌드 확인** — `npm run build` 또는 해당 빌드 명령 실행
2. **UI 깨짐 확인** — 수정한 컴포넌트와 연관 화면 체크
3. **기능 동작 확인** — 수정이 기존 기능을 깨뜨리지 않았는지
4. **문서 일관성** — 수정된 코드와 문서가 일치하는지

**검증 실패 시**: 실패 원인 분석 → 추가 수정 → 재검증 (최대 3회)

#### Step 6: 수정 로그 기록

변경 사항을 기획서 하단 변경 이력에 기록:

```markdown
| v1.x | {오늘 날짜} | fix: {수정 요약} — 영향: {변경된 파일 수}개 파일 |
```

#### Step 7: Manager Memory 기록

Tech Lead 실행 완료 후 Manager가 memory에 기록:
- **change**: 수정 내용, 영향 범위, 체이닝 경로
- **dependency**: 새로 발견된 피처 간 의존성 (있는 경우)
- **debt**: 임시 처리된 부분, 추후 개선 필요사항 (있는 경우)
- **decision**: 정책 변경 등 의사결정 (있는 경우)
