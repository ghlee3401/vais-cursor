### 🔎 check — 빌드 검증 + Gap 분석

**빌드/실행 검증 → Gap 분석 → 보안 스캔** 순서로 진행합니다.

#### Step 1: 빌드/실행 검증

1. 의존성 설치 확인 (`npm install` 등)
2. 빌드 확인 (`npm run build` 등)
3. 서버 시작 확인 (dev server 기동)
4. 핵심 동작 확인 (메인 페이지 로드, API 응답)
5. **빌드 실패 시**: 에러 분석 → 수정 시도. Gap 분석으로 넘어가지 않음.

#### Step 2: Gap 분석

1. **피처 레지스트리 로드** (`.vais/features/$1.json`) — plan에서 정의된 기능 목록을 기준으로 Gap 분석
2. **설계 문서 수집**: plan, design, design-db 문서 읽기
3. **구현 코드 수집**: 해당 피처 코드 전체 읽기
4. **자동 비교**: 레지스트리의 `features[]` 각 항목 vs 코드 구현 여부. 구현된 기능은 레지스트리 `status`를 `"완료"`로 업데이트
5. **일치율**: `(구현된 수 / 전체 수) x 100` — 레지스트리 기준
5. **Gap은 패치 단위로 출력**:
   ```markdown
   | # | 미구현 항목 | 출처 | 수정 대상 파일 | 수정 범위 | 수정 내용 |
   |---|-----------|------|-------------|---------|---------|
   ```
6. **Gap 방향 판단**: 각 Gap 항목에 대해 AskUserQuestion으로 선택 요청 — "구현을 설계에 맞출지 / 설계를 구현에 맞출지". 사용자 선택에 따라 코드 수정 또는 설계 문서 업데이트
7. **자동 반복**: 90% 미만 → 미구현 항목 자동 구현 시도 (최대 5회)
7. `docs/06-check/$1.md`에 저장

#### Step 3: 보안 스캔

1. OWASP Top 10 체크리스트 기반 검사
2. 인증/인가, SQL Injection, XSS, CSRF 점검
3. 결과를 Gap 분석 문서에 포함

#### Step 4: QA 시나리오 리스트 생성

1. **기획서(plan) 읽기** → 기능 요구사항 추출
2. **핵심 기능 시나리오**: 각 요구사항별 사용자 관점 테스트 시나리오 작성
   - 시나리오명, 사전 조건, 테스트 단계, 기대 결과, 우선순위(P0/P1/P2)
3. **엣지 케이스**: 빈 입력, 경계값, 권한 없음, 네트워크 오류 등 예외 상황
4. **UI/UX 검증**: 반응형, 로딩 상태, 에러 상태, 빈 상태
5. QA 리스트를 `docs/06-check/$1.md`에 포함 → review 단계에서 판정
